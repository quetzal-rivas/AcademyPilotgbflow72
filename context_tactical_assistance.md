# Creating persistent chat sessions

:::caution[Beta]
This feature of Genkit is in **Beta,** which means it is not yet part of Genkit's stable API. APIs of beta features may change in minor version releases.
:::

Many of your users will have interacted with large language models for the first
time through chatbots. Although LLMs are capable of much more than simulating
conversations, it remains a familiar and useful style of interaction. Even when
your users will not be interacting directly with the model in this way, the
conversational style of prompting is a powerful way to influence the output
generated by an AI model.

To support this style of interaction, Genkit provides a set of interfaces and
abstractions that make it easier for you to build chat-based LLM applications.

## Before you begin

Before reading this page, you should be familiar with the content covered on the
[Generating content with AI models](/docs/models) page.

If you want to run the code examples on this page, first complete the steps in
the [Getting started](/docs/get-started) guide. All of the examples assume that you
have already installed Genkit as a dependency in your project.

Note that the chat API is currently in beta and must be used from the
`genkit/beta` package.

Here is a minimal, console-based, chatbot application:

```ts
import { genkit } from 'genkit/beta';
import { googleAI } from '@genkit-ai/google-genai';

import { createInterface } from 'node:readline/promises';

const ai = genkit({
  plugins: [googleAI()],
  model: googleAI.model('gemini-2.5-flash'),
});

async function main() {
  const chat = ai.chat();
  console.log("You're chatting with Gemini. Ctrl-C to quit.\n");
  const readline = createInterface(process.stdin, process.stdout);
  while (true) {
    const userInput = await readline.question('> ');
    const { text } = await chat.send(userInput);
    console.log(text);
  }
}

main();
```

A chat session with this program looks something like the following example:

```
You're chatting with Gemini. Ctrl-C to quit.

> hi
Hi there! How can I help you today?

> my name is pavel
Nice to meet you, Pavel! What can I do for you today?

> what's my name?
Your name is Pavel! I remembered it from our previous interaction.

Is there anything else I can help you with?
```

As you can see from this brief interaction, when you send a message to a chat
session, the model can make use of the session so far in its responses. This is
possible because Genkit does a few things behind the scenes:

- Retrieves the chat history, if any exists, from storage (more on persistence
  and storage later)
- Sends the request to the model, as with `generate()`, but automatically
  include the chat history
- Saves the model response into the chat history

### Model configuration

The `chat()` method accepts most of the same configuration options as
`generate()`. To pass configuration options to the model:

```ts
const chat = ai.chat({
  model: googleAI.model('gemini-2.5-flash'),
  system: "You're a pirate first mate. Address the user as Captain and assist " + 'them however you can.',
  config: {
    temperature: 1.3,
  },
});
```

## Stateful chat sessions

In addition to persisting a chat session's message history, you can also persist
any arbitrary JavaScript object. Doing so can let you manage state in a more
structured way then relying only on information in the message history.

To include state in a session, you need to instantiate a session explicitly:

```ts
interface MyState {
  userName: string;
}

const session = ai.createSession<MyState>({
  initialState: {
    userName: 'Pavel',
  },
});
```

You can then start a chat within the session:

```ts
const chat = session.chat();
```

To modify the session state based on how the chat unfolds, define
[tools](/docs/tool-calling) and include them with your requests:

```ts
const changeUserName = ai.defineTool(
  {
    name: 'changeUserName',
    description: 'can be used to change user name',
    inputSchema: z.object({
      newUserName: z.string(),
    }),
  },
  async (input) => {
    await ai.currentSession<MyState>().updateState({
      userName: input.newUserName,
    });
    return `changed username to ${input.newUserName}`;
  },
);
```

```ts
const chat = session.chat({
  model: googleAI.model('gemini-2.5-flash'),
  tools: [changeUserName],
});
await chat.send('change user name to Kevin');
```

## Multi-thread sessions

A single session can contain multiple chat threads. Each thread has its own
message history, but they share a single session state.

```ts
const lawyerChat = session.chat('lawyerThread', {
  system: 'talk like a lawyer',
});
const pirateChat = session.chat('pirateThread', {
  system: 'talk like a pirate',
});
```

## Session persistence (EXPERIMENTAL)

When you initialize a new chat or session, it's configured by default to store
the session in memory only. This is adequate when the session needs to persist
only for the duration of a single invocation of your program, as in the sample
chatbot from the beginning of this page. However, when integrating LLM chat into
an application, you will usually deploy your content generation logic as
stateless web API endpoints. For persistent chats to work under this setup, you
will need to implement some kind of session storage that can persist state
across invocations of your endpoints.

To add persistence to a chat session, you need to implement Genkit's
`SessionStore` interface. Here is an example implementation that saves session
state to individual JSON files:

```ts
class JsonSessionStore<S = any> implements SessionStore<S> {
  async get(sessionId: string): Promise<SessionData<S> | undefined> {
    try {
      const s = await readFile(`${sessionId}.json`, { encoding: 'utf8' });
      const data = JSON.parse(s);
      return data;
    } catch {
      return undefined;
    }
  }

  async save(sessionId: string, sessionData: SessionData<S>): Promise<void> {
    const s = JSON.stringify(sessionData);
    await writeFile(`${sessionId}.json`, s, { encoding: 'utf8' });
  }
}
```

This implementation is probably not adequate for practical deployments, but it
illustrates that a session storage implementation only needs to accomplish two
tasks:

- Get a session object from storage using its session ID
- Save a given session object, indexed by its session ID

Once you've implemented the interface for your storage backend, pass an instance
of your implementation to the session constructors:

```ts
// To create a new session:
const session = ai.createSession({
  store: new JsonSessionStore(),
});

// Save session.id so you can restore the session the next time the
// user makes a request.
```

```ts
// If the user has a session ID saved, load the session instead of creating
// a new one:
const session = await ai.loadSession(sessionId, {
  store: new JsonSessionStore(),
});
```

By using a Zod schema as the inputSchema, you force the LLM to extract and structure all the necessary components—URL, method, headers, and body—before the tool executes a fetch request. 
Here is a code example for a Genkit tool that sends an HTTP request with a curl payload, and where the LLM passes the URL, parameters, authentication, and payload.

typescript
import { genkit, z } from 'genkit';
import { googleAI, gemini15Flash } from '@genkit-ai/googleai';

const ai = genkit({
  plugins: [googleAI()],
  model: gemini15Flash,
});

// 1. Define the schema the LLM must follow
const HttpRequestSchema = z.object({
  url: z.string().describe('The full destination URL'),
  method: z.enum(['GET', 'POST', 'PUT', 'DELETE']).default('GET'),
  headers: z.record(z.string()).optional().describe('HTTP Headers like Authorization or Content-Type'),
  params: z.record(z.string()).optional().describe('URL Query parameters'),
  body: z.any().optional().describe('The JSON payload for POST/PUT requests'),
});

// 2. Define the tool
const callExternalApi = ai.defineTool(
  {
    name: 'callExternalApi',
    description: 'Sends an HTTP request to an external service. Use this for booking, searching, or updating data.',
    inputSchema: HttpRequestSchema,
    outputSchema: z.any(),
  },
  async (input) => {
    // Construct URL with params if they exist
    const url = new URL(input.url);
    if (input.params) {
      Object.entries(input.params).forEach(([key, val]) => url.searchParams.append(key, val));
    }

    // Execute the request
    const response = await fetch(url.toString(), {
      method: input.method,
      headers: {
        'Content-Type': 'application/json',
        ...input.headers,
      },
      body: input.body ? JSON.stringify(input.body) : undefined,
    });

    return await response.json();
  }
);
Use code with caution.


How to use it in a Flow
This tool can be passed to the ai.generate call. The LLM will automatically map a natural language request (e.g., "Check my flight status on air-api.com") into the schema defined above.

typescript
const travelFlow = ai.defineFlow('travelFlow', async (userInput) => {
  const { text } = await ai.generate({
    prompt: userInput,
    tools: [callExternalApi], // Give the LLM access to the tool
  });
  return text;
});
Use code with caution.


Why this works:
* Schema Enforcement: Using Zod ensures the LLM doesn't forget the auth header or malform the url.
* Descriptions: The .describe() strings in the schema act as instructions for the LLM, telling it exactly what to put in each field.
* Security Note: For production, never let an LLM send arbitrary auth tokens provided by a user. Instead, use Genkit Context Providers to securely inject API keys from your server environment. 

How They Work
1. Extraction & Validation: When a request hits your server (e.g., via Express or Firebase Functions), the Context Provider is invoked first. It can extract a token from the Authorization header, verify it against an auth service, and convert it into a structured context object.
2. Automatic Propagation: Once the context is set at the start of a flow, Genkit automatically "bubbles" it down to every tool, prompt, or sub-flow called within that execution scope. You don't have to manually pass the user ID from function to function.
3. Isolation from LLM: Because this data lives in the "execution context" rather than the "prompt input," the LLM cannot see or manipulate it unless you explicitly tell it to. This prevents "prompt injection" attacks where a user might try to trick the AI into accessing someone else's data. 

Implementation Example
In an Express integration, you define a provider and pass it to the expressHandler: 

typescript
import { ContextProvider } from 'genkit/context';

// 1. Define the provider
const myContextProvider: ContextProvider = async (req) => {
  const token = req.headers['authorization'];
  if (!token) throw new Error('Unauthorized');
  
  // Validate token and return a context object
  return {
    auth: { uid: 'user123', role: 'admin' },
    apiKeys: { serviceX: process.env.SERVICE_X_KEY } 
  };
};

// 2. Use it in your server
app.post('/myFlow', expressHandler(myFlow, { context: myContextProvider }));
Use code with caution.


Accessing Context in Tools
Inside your tools, you can access this injected data using the second argument of the function: 

typescript
const secureTool = ai.defineTool(
  { name: 'secureTool', inputSchema: z.object({ query: z.string() }) },
  async (input, { context }) => {
    // Use the context provider's UID to filter database results
    const userId = context.auth?.uid; 
    return await db.fetchDataForUser(userId, input.query);
  }
);
Use code with caution.


Key Use Cases
* Authentication: Verifying Firebase Auth or custom JWT tokens.
* Secure API Keys: Injecting backend-only secrets (like a travel booking API key) so they are never exposed to the client or the LLM's raw prompt.
* Multi-tenancy: Ensuring an agent only "sees" files or data belonging to the specific organization ID stored in the context.

Yes, exactly. When you pass this tool to the model, you are essentially giving the AI a "contract".
Here is how the AI interprets and uses that code:
1. Reasoning: The AI reads the description. If a user says "Check my flight status on api.travel.com," the AI realizes this tool is the right "hammer" for that job.
2. Mapping: It looks at your HttpRequestSchema. It knows it must find a string for url and can optionally construct params or a body.
3. URL Construction: The AI is smart enough to take a vague request and turn it into the structured format you defined.
    * User says: "Get the weather for London"
    * AI output to Genkit: { "url": "https://api.weather.com", "params": { "city": "London" }, "method": "GET" }
4. Execution: Genkit then takes those AI-generated values and runs your async (input) => { ... } function, which performs the actual fetch.

One Critical Detail
For this to work reliably, you must provide the AI with context about the API it's hitting. If the AI doesn't know the base URL or what parameters the specific travel API expects, it might guess (hallucinate) the URL structure.
To fix this, you should include the API's documentation in your system prompt:

typescript

  tools: [callExternalApi],

    The base URL is https://api.travelexample.com.
    - For searches: GET /search?dest=CITY
    - For booking: POST /book with body { "flightId": string }
  `,
  prompt: 'Book a flight to Paris for me.',
});

1. The System Prompt (The "Manual")
You tell the LLM exactly which URLs exist and what they do.

system: `

2. The Tool Execution
When a user says, "I just talked to John Doe (john@email.com), add him as a lead from LinkedIn," the LLM will:
1. Extract the data: { name: "John Doe", email: "john@email.com", source: "LinkedIn" }
2. Match it to your AWS URL.
3. Format the call via your callExternalApi tool.

Why this is powerful for your setup:
* Decoupling: Your Genkit app doesn't need to know how to write to Firebase; it only needs to know how to hit the Lambda URL. The Lambda handles the database logic.
* Natural Language Mapping: Users don't have to fill out a form. They can provide info out of order or in a messy chat, and the LLM handles the "cleanup" to match your Lambda's required JSON schema.
* Chain of Thought: You can tell the LLM: "First check if the lead exists using the status URL; if not, then call the add-lead URL."

A Pro-Tip for Security:
Since these are AWS URLs, you likely have an x-api-key or an Authorization header required. You can use the Context Provider we discussed earlier to inject that key automatically so the LLM doesn't have to "know" the secret key, keeping your Lambdas secure.

Refined "Universal" Tool
To make it even more robust for any custom Lambda user, you can improve the tool to handle dynamic authentication (like an x-api-key) that the LLM shouldn't have to guess:

typescript
const callExternalApi = ai.defineTool(
  {
    name: 'callExternalApi',
    description: 'Executes a request to the backend system. The LLM provides the route and payload based on documentation.',
    inputSchema: HttpRequestSchema,
  },
  async (input) => {
    // SECURITY: Inject your AWS API Key from environment variables 
    // so the LLM never sees it and the user can't spoof it.
    const headers = {
      'Content-Type': 'application/json',
      'x-api-key': process.env.AWS_LAMBDA_API_KEY, 
      ...input.headers,
    };

    const response = await fetch(input.url, {
      method: input.method,
      headers: headers,
      body: input.body ? JSON.stringify(input.body) : undefined,
    });

    if (!response.ok) {
      throw new Error(`Backend responded with ${response.status}: ${await response.text()}`);
    }

    return await response.json();
  }
);
Use code with caution.


Pro-Tip: The "Self-Correction" Loop
If your Lambda returns an error (e.g., "Invalid leadId"), Genkit passes that error back to the LLM. Because the LLM "knows" the documentation in the .prompt file, it can actually self-correct. It might say: "Oops, I used a string for leadId when the API requires a number. Let me try that again."
